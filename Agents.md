**Agents Policy — "No AI Slop"**

Purpose
- Provide a clear, enforceable policy for any automated agent (human, CI, or AI) that may edit this repository.
- Prevent low-quality or unsafe AI-generated changes ("AI slop").

Scope
- Applies to all source, shell scripts, installer scripts, and build/test helpers in this repository. See the "Critical files" section below.

Rules for any Agent (human or AI)
1. Confirm intent: Do not edit files without an explicit task and approval. If uncertain, open an issue instead of guessing.
2. Verify syntax, style, and canonical patterns from authoritative sources before editing:
   - For shell scripts consult shellcheck rules and POSIX / Bash / Zsh manpages.
   - For Node.js/JS consult Node.js docs and ESLint rules.
   - For other languages consult their official docs and linters.
3. No AI Slop: Do not commit code that appears to be low-quality, hallucinated, or not verified by running appropriate linters and tests.
4. Human-style code: Write code in clear, idiomatic style consistent with the repository. Avoid overly terse or unnatural patterns that indicate wholesale autogenerated output.
5. Comments and documentation:
   - For every function (or top-level command block) add a one-line comment immediately above describing: purpose, inputs, outputs, and typical usage (single concise line).
   - For any non-trivial or safety-sensitive line, add an inline or adjacent one-line comment explaining intent and edge cases.
   - Do NOT add redundant comments for trivial lines (e.g., `x=1` does not need "set x to 1"). Focus on intent and invariants.
6. Source and citations: If the agent used an external reference (web page, manpage, blog), include a short citation comment linking the authoritative source (one-line) in the commit message or PR.
7. Verification required BEFORE commit:
   - Shell scripts: run `shellcheck` and `bash -n` or `zsh -n` as applicable.
   - Zsh files: run `zsh -n` and a full interactive smoke test where appropriate.
   - JS files: run `eslint --fix` and unit checks where available.
   - Run `bin/test_repo.sh` and fix any reported failures.
8. Testing and idempotence: Installer scripts should be idempotent and not remove user data silently. Any destructive action must be guarded and documented.
9. Safety for network operations: Do not pipe remote scripts directly into shell (no `curl | sh`). Instead, download, verify checksum, and run after review. Document the checksum source.
10. Commits and PRs: Make small, atomic commits with descriptive messages. Include verification steps in the PR description and link to the linter/test output.

Critical files (must never contain AI slop)
- install.sh
- install/install.sh
- install/uninstall.sh
- install/fonts-install.sh
- core/*.sh (system-detect.sh, ssh-agent.sh, tools.sh, logging.sh, battery.sh)
- shells/* (all shell startup files: `.zshrc`, `.zprofile`, `modes/*.zsh`, `functions*.zsh`, bash and fish equivalents)
- bin/*.sh (setup, test, dotupdate, dotupdate_bg, test_repo, setup_hooks)
- any scripts with shebangs or installer behavior

Suggested enforcement workflow
1. Any agent must open a draft PR with changes and the verification steps listed in the PR description.
2. CI must run `shellcheck`, `zsh -n` for zsh files, `eslint` for JS, and `bin/test_repo.sh`.
3. If CI passes and a human reviewer confirms intent and comment quality, merge.

Examples of unacceptable output (AI slop)
- Unverified network commands that pipe to shell (`curl ... | sh`).
- Missing function-level documentation or one-line usage comments for installers and core functions.
- Overly generic comments like "fix bugs" without explanation of scenario or tradeoffs.

How to fix violations
- Reproduce the failure locally, add or improve comments as required, run linters and tests, and push a follow-up commit referencing the issue.

Minimal checklist for reviewers
- Does the change include one-line usage comment for each new function? Yes/No
- Were linters run and errors fixed? Yes/No (attach output)
- Did the author cite authoritative references if using web guidance? Yes/No
- Is there any network download or privilege escalation? If yes, has it been justified and made safe? Yes/No

If you are an AI: follow these exact steps before producing a code change
1. Identify the file(s) to change and explain (one sentence) why the change is necessary.
2. Search authoritative docs for syntax and best practice; cite the sources in the PR.
3. Produce a small patch (single responsibility) and include one-line function usage comments.
4. Run the repo verification steps locally (linters/tests) and paste their outputs in the PR.
5. Do not auto-merge. Request a human review.

Contact / Escalation
- If unsure whether a change is safe, open an issue with the label `security` or `installer` and request human review. Do NOT push directly to main.
